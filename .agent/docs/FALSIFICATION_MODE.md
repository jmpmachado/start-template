# Falsification Mode for AI Agents

## Theoretical basis and relationship to the Operating Rules

This document consolidates the theoretical and bibliographic foundation for using **falsifiability as the default operational mode** in AI agents, and shows how it connects to the Operating Rules — in particular:

- Rule 8 — **Analysis mode — falsification by default**
- Rule 9 — **Code artifact modes — explicit trigger only**
- Rule 10 — **Quality by Execution Mode**
- Rule 11 — **Minimal Traceable Memory**

---

## 1. Overview

The agent system adopts **falsification by default** for analysis, review, and audit tasks: gaps and failures are assumed to exist, the agent actively seeks evidence of invariant violations, and each finding is qualified by severity (🔴 / 🟡 / 🟢) and declared scope.

This approach rests on four main theoretical pillars:

- The Popperian tradition of **falsifiability** as the criterion for rigorous hypothesis testing.
- Recent agentic frameworks that implement **falsification-first** as an operational mechanism (e.g., POPPER).
- Proposals for **artificial epistemic agents** that treat AI agents as epistemic entities, emphasising "robust falsifiability".
- **Agentic AI** patterns (Andrew Ng) that cleanly separate construction from self-criticism (Reflection).

---

## 2. POPPER: sequential falsification agent

The **POPPER** framework (Automated Hypothesis Validation with Agentic Sequential Falsifications) proposes an agent specialised in validating hypotheses through **sequential falsifications**, explicitly inspired by Karl Popper.

### 2.1 Core idea

- Instead of trying to **confirm** a hypothesis, POPPER tests it through **experiments designed to refute it**.
- Hypotheses (including those generated by LLMs) are decomposed into falsifiable sub-claims, and the system designs experiments that maximise the chance of finding empirical contradictions.

### 2.2 Relevant architecture

- **Experiment-design agent**: generates test scenarios aimed at maximising the chance of refutation.
- **Execution agent**: collects data, runs tests, and rigorously controls type-I error via sequential statistical methods (e-values, sequential testing).

### 2.3 Connection to the rules

| POPPER element | Corresponding rule |
| :--- | :--- |
| Test hypotheses by seeking refutations, not confirmations | Rule 8: "analysis mode — falsification by default" |
| Decompose hypotheses into testable sub-claims | Rule 8: "assume gaps exist, probe operational completeness" |
| Rigorous type-I error control in sequential tests | Rule 10: "Quality by Execution Mode… full-falsify for high-risk changes" |

The **`*-falsify` mode** is conceptually analogous: rather than just checking surface conformance (lint), the agent seeks scenarios that **break invariants, contracts, and assumptions** systematically.

## 3. Epistemic agents and "robust falsifiability"

The paper **Architecting Trust in Artificial Epistemic Agents** discusses LLMs and AI agents as **epistemic agents**: systems that pursue epistemic goals (knowing, justifying, updating beliefs) and therefore need to be **epistemically trustworthy**.

### 3.1 Epistemic properties

The paper proposes that trustworthy agents must demonstrate, among other things:

- **Epistemic competence** (understanding and managing uncertainty, evidence, and belief revision).
- **Robust falsifiability** — the agent's hypotheses and belief states must be testable and revisable when confronted with counter-evidence.
- **Virtuous epistemic behaviours** (honesty about uncertainty, willingness to self-correct, avoidance of epistemic manipulation).

### 3.2 Connection to the rules

| Concept | Implementation in the rule set |
| :--- | :--- |
| Robust falsifiability of the agent's beliefs | Rule 8: agent assumes gaps and failures exist and actively tries to refute them |
| Epistemic competence and transparency | Rule 8: requires the agent to declare `Mode` and `artifact`, plus the `Scope` checked |
| Verification and audit infrastructure | Rules 7 and 11: prohibition on silent `CLASS_MAP` edits and traceable operational memory |

By requiring the agent to **declare mode (`construct`, `lint`, `falsify`) and domain (`code`, `doc`, `map`, `machine`) at the start of every analysis turn**, the template translates the abstract requirement of "robust falsifiability" into concrete, auditable operational discipline.

## 4. Agentic AI patterns (Andrew Ng)

The **Agentic AI design patterns** proposed by Andrew Ng highlight four main patterns: **Reflection, Tool Use, Planning, and Multi-agent collaboration**.

### 4.1 Reflection pattern

- In the **Reflection** pattern, the agent generates an initial result, **critiques its own output for errors and gaps**, and only then produces a revised version.
- This self-critique functions as a local falsification test: the agent tries to refute its own response before publishing it.

### 4.2 Connection to the rules

| Pattern | Corresponding rule |
| :--- | :--- |
| Reflection: self-criticism and iterative revision | Rule 8: "Analysis mode — falsification by default" |
| Separation between production and verification | Rule 9: `*-construct` vs `*-lint` vs `*-falsify` |
| Human-in-the-loop for risky tasks | Rule 6: destructive operations require explicit human confirmation |

The template codifies Reflection as an **explicit mode** (`*-falsify`) rather than an implicit heuristic: this avoids ambiguity and forces the agent to declare when it is building and when it is trying to break.

## 5. Error statistics and test severity

The **error statistics** tradition (e.g., Deborah Mayo) articulates the idea of **severe testing**: hypotheses are only corroborated when they pass tests that had a high capacity to reveal their falsity had they actually been false.

### 5.1 Key concepts

- **Deductive falsification**: if hypothesis H implies observation O and we observe ¬O, we infer ¬H (or that something in the theoretical system surrounding H is wrong).
- **Test severity**: a test is severe if it had a good chance of revealing an error, should the hypothesis be false; weak tests do not justify high confidence.

### 5.2 Connection to the rules

| Statistical concept | Implementation |
| :--- | :--- |
| Severe vs superficial tests | Rule 9: `*-lint` (static conformance check) vs `*-falsify` (adversarial — try to break) |
| Different levels of rigour proportional to risk | Rule 10: `lint-all` for routine changes, `full-falsify` for high-risk changes |
| Documenting test scope | Rule 8: "Scope: \<files or sections checked\>" is analogous to reporting experimental design and reach |

## 6. Direct mapping: Rules 8 and 9

### 6.1 Rule 8 — Analysis mode: falsification by default

Rule 8 establishes that all analysis, review, or audit operates in **falsification mode**: the agent does not ask "does this look correct?" but instead "what would have to be true for this to be wrong?".

Operationally, this imposes three obligations before any analysis output:

1. **Declare the mode**: `> Mode: falsify | artifact: doc` (or `code`, `map`, `machine`).
2. **Assume gaps by default**: the agent does not wait to find problems — it presupposes they exist and looks for evidence of their absence.
3. **Qualify each finding**: 🔴 critical (affects correctness, security, compliance, or irreversibility) / 🟡 operational (affects functionality or maintainability) / 🟢 observation (improvement with no immediate impact).
4. **Declare scope**: `Scope: <files or sections checked>` — analogous to experimental-design reporting in statistics.

The separation between `construct` and `falsify` is cognitively fundamental: building and auditing are opposite processes. Conflating them causes confirmation bias — the builder tends to validate their own work rather than attack it.

### 6.2 Rule 9 — Code artifact modes: explicit trigger only

Rule 9 extends the mode system to the artifact level, introducing four domains (`code`, `doc`, `map`, `machine`) and three modes (`construct`, `lint`, `falsify`).

| Mode | Intent | Theoretical analogy |
| :--- | :--- | :--- |
| `*-construct` | Build or edit; no audit | Hypothesis-generation phase |
| `*-lint` | Static conformance check (structure, format, obvious omissions) | Weak test (syntactic consistency verification) |
| `*-falsify` | Adversarial: enumerate breaking inputs, unreachable states, contract violations | Severe test (Mayo); sequential falsification (POPPER) |

The explicit trigger is essential: without it, the agent could switch between modes without signalling, introducing epistemic inconsistency. The declaration requirement is analogous to **pre-registration** in experimental science — the investigator commits to the test design before seeing the data.

**Ambiguity resolution** (no pause, in order):

1. File extension: `.py/.ts/.js` → `code`; `.md` → `doc`; `.yaml` in `rfcs/` or `.agent/context/` → `doc`; `.yaml` in `infra/` or workflow → `code`.
2. Location: `infra/scripts/` → `code`; `.agent/context/` → `doc`.
3. Default: `doc`.

## 7. Conclusion: Rules 8–9 as operationalisation of scientific falsification

Rules 8 and 9 form a coherent system that translates three theoretical traditions into operational discipline:

| Tradition | Contribution | Operationalisation in the rules |
| :--- | :--- | :--- |
| Popper / POPPER framework | Falsification as the primary testing mechanism | Rule 8: falsify by default; active search for refutations |
| Epistemic agents (Marchal et al.) | Robust falsifiability and epistemic transparency | Rule 8: mode + scope declaration; Rule 9: explicit trigger |
| Agentic AI / Reflection (Ng) | Separation of construction and self-criticism | Rule 9: `*-construct` vs `*-falsify` are distinct modes, not implicit phases |
| Error statistics (Mayo) | Severe tests proportional to risk | Rule 10: routine `lint-all` vs `full-falsify` for high-risk changes |

The practical result is that any agent following Rules 8–9 is, knowingly or not, implementing a **sequential severe-testing protocol** applied to software artifacts and documentation — with traceability guaranteed by Rule 11.

## 8. Rules 10 and 11: execution and memory

### 8.1 Rule 10 — Quality by Execution Mode

Rule 10 establishes that, before concluding any delivery, an explicit **verification mode** appropriate to the risk must be chosen:

- `lint-all` for routine changes.
- `full-falsify` for high-risk changes (release, post-incident, changes to critical invariants).

This operationalises the idea that **severe tests** should be reserved for hypotheses or changes whose failure would be costly, as argued in the error-statistics tradition and in sequential-falsification frameworks.

### 8.2 Rule 11 — Minimal Traceable Memory

Rule 11 defines that a **minimal, traceable operational memory** must exist in `.agent/MEMORY.md`, with structured outputs that preserve:

- **Severity** (🔴 / 🟡 / 🟢).
- **Scope** (files, sections, and modes verified).
- Sufficient context for agent handoff without rework.

This connects to recent work on **agent memory**, which emphasises compact, structured records for traceability, and to the **epistemic-agents** literature, which requires standardised, auditable logging for reconstructing decisions and failures.

### 8.3 Example MEMORY.md entry

```markdown
| a1b2c3d4 | 2026-05-29 | high | .agent/context/ARCHITECTURE.md | falsify: 3 invariants violated (layers 2–4 missing dependency contracts) | abc1234 |
```

Each column corresponds to: `id` · `timestamp` · `severity` · `scope` · `summary (mode + finding)` · `source_commit`.

The `summary` field should include the **mode** used (`falsify`, `lint`, `construct`) as a prefix — this preserves in the audit trail which type of test was applied, making the memory retroactively falsifiable: a future auditor can verify whether the declared mode was appropriate for the risk of the change.
